Aptos Explorer — Vercel deployment setup

Steps to stand up the explorer on Vercel alongside the existing Netlify deployment. Netlify keeps serving production until you explicitly cut DNS over.

What the repo already has Code changes are in place: vercel.ts + a build script (scripts/build-vercel-output.mjs) that emits Vercel's Build Output API v3 layout, plus VITE_VERCEL_CONTEXT wiring for preview-deploy API key suppression. Nothing about Netlify changed. See AGENTS.md → "Vercel and Netlify dual deploy" for the parity rules.

1. Create the Vercel project

  1. In the Vercel dashboard, click Add New… → Project and import the aptos-explorer GitHub repo.
  2. On the configuration screen:
    • Framework Preset: Other
    • Build Command: leave blank — vercel.ts sets buildCommand: "node scripts/build-vercel-output.mjs".
    • Output Directory: leave blank — vercel.ts sets outputDirectory: ".vercel/output".
    • Install Command: leave blank — vercel.ts sets installCommand: "pnpm install --frozen-lockfile".
    • Node.js Version: 24.x (or higher). The SSR function pins nodejs24.x; the build itself should run on the same major.
  3. Do not deploy yet — environment variables need to be in place first.

2. Set environment variables

Mirror everything from Netlify. Set scope to Production, Preview, and Development unless noted.

Aptos API keys (required)

Each network needs both a client-side VITE_* key (baked into the browser bundle at build) and a server-side key (read by SSR at runtime). The values should be identical per network so browser and server share one identity / rate-limit bucket.

Client (browser bundle) Server (SSR)
VITE_APTOS_MAINNET_API_KEY APTOS_MAINNET_API_KEY
VITE_APTOS_TESTNET_API_KEY APTOS_TESTNET_API_KEY
VITE_APTOS_DEVNET_API_KEY APTOS_DEVNET_API_KEY
Preview deploys The keys are scoped to explorer.aptoslabs.com, so Vercel preview URLs (*.vercel.app) will hit rate limits. That's expected — the client code already detects VITE_VERCEL_CONTEXT === "preview" and suppresses keys so the browser falls back to anonymous rate limits with a loud console warning, matching Netlify's deploy-preview behavior.

Optional: analytics & feature flags

VariablePurpose
REACT_APP_GTM_ID Google Tag Manager ID (Production only).
VITE_FEATURE_NAME Feature tier banner — prod / dev / earlydev. Defaults to prod.
VITE_VALIDATOR_STATS_CACHE_VERSION Cache-bust string for validator stats.
VITE_GITHUB_TOKEN Higher rate limit on GitHub-source queries (optional).

Auto-injected (do not set manually)

  • VERCEL_ENV — Vercel sets this to production/preview/development. The build script reads it and exports VITE_VERCEL_CONTEXT for the client bundle.

3. Trigger the first preview deploy

  1. Click Deploy. Vercel will run pnpm install --frozen-lockfile, then node scripts/build-vercel-output.mjs (which itself runs pnpm routes:generate + vite build + emits .vercel/output/).
  2. Expected build time: roughly the same as Netlify — Vite build dominates.
  3. Once green, open the preview URL and watch the Functions tab in the deployment view to confirm the SSR function (index.func) is invoked on page loads.

4. Parity checklist

Before considering Vercel ready for production, validate parity with a few spot checks. Replace {preview} with the Vercel URL.

SSR cache headers

curl -sI https://{preview}/ | grep -i cache-control
# expect:  Cache-Control: public, max-age=0, s-maxage=15, stale-while-revalidate=60

curl -sI https://{preview}/txn/1 | grep -i cache-control
# expect:  Cache-Control: public, max-age=0, s-maxage=3600, stale-while-revalidate=86400

curl -sI https://{preview}/account/0x1 | grep -i cache-control
# expect:  Cache-Control: public, max-age=0, s-maxage=30, stale-while-revalidate=120

Security & agent-discovery headers

curl -sI https://{preview}/ | grep -iE 'x-frame|x-content|referrer|permissions|link|vary'
# expect: X-Frame-Options, X-Content-Type-Options, Referrer-Policy,
#         Permissions-Policy, Link (api-catalog, agent-skills, llms, sitemap),
#         and Vary: Accept

Well-known files

curl -sI https://{preview}/.well-known/api-catalog | grep -iE 'content-type|access-control'
# expect: Content-Type: application/linkset+json
#         Access-Control-Allow-Origin: *

curl -sI https://{preview}/llms.txt | grep -iE 'content-type|cache-control|x-robots-tag'
# expect: Content-Type: text/plain; charset=utf-8
#         Cache-Control: public, max-age=86400
#         X-Robots-Tag: noarchive

Markdown homepage negotiation

curl -s -H 'Accept: text/markdown' https://{preview}/ | head -3
# expect: the contents of public/llms.txt (not the SSR HTML)

Legacy host redirects (only verifiable after DNS)

# Once you point explorer.devnet.aptos.dev at Vercel:
curl -sI https://explorer.devnet.aptos.dev/account/0x1
# expect: 301 → https://explorer.aptoslabs.com/account/0x1?network=devnet

5. Hook up a stable preview hostname (recommended)

Add a CNAME like explorer-vercel.aptoslabs.com → Vercel for ongoing parity validation while Netlify keeps the canonical traffic. In the Vercel project: Settings → Domains → Add.

6. Production cutover (when ready)

  1. Run the parity checklist on a Vercel preview that's been live for at least a few days.
  2. Move explorer.aptoslabs.com (and the legacy *.aptos.dev subdomains) DNS records over to Vercel.
  3. Keep Netlify running for at least a week as a fallback — both configs (netlify.toml and vercel.ts) must stay in sync until then. See AGENTS.md for the parity rules.
  4. Once you're confident, file a separate PR to remove Netlify configuration: netlify.toml, @netlify/vite-plugin-tanstack-start in vite.config.ts, the VITE_NETLIFY_CONTEXT type, and the half of isHostedPreview that checks it.

Troubleshooting

"y.handler is not a function"

Means the SSR function's default export wasn't a callable. The wrapper at .vercel/output/functions/index.func/index.mjs must be export default ssr.fetch; — if you see this error, confirm dist/server/ssr.js still exports default { fetch } (the same contract Netlify relies on).

API keys leak into preview URLs

Confirm the build log printed something like VITE_VERCEL_CONTEXT=preview. If not, the build script didn't see VERCEL_ENV — open an issue and check scripts/build-vercel-output.mjs.

504 timeouts on slow pages

Unlikely — the SSR function is configured for maxDuration: 300. Active CPU billing means longer timeouts are essentially free unless the handler is CPU-bound, which TanStack Start's SSR isn't. If you do see 504s, the upstream Aptos API is the first place to look.

Files involved

  • vercel.ts — Vercel project config (build command, install command, output directory; no framework preset).
  • scripts/build-vercel-output.mjs — runs pnpm run build, then emits .vercel/output/ (static assets, SSR function, routes / headers / redirects).
  • .vercelignore — keeps Netlify output out of the upload.
  • app/lib/constants.ts — isHostedPreview checks both VITE_NETLIFY_CONTEXT and VITE_VERCEL_CONTEXT.
  • app/types/declarations.d.ts — TypeScript type for VITE_VERCEL_CONTEXT.
